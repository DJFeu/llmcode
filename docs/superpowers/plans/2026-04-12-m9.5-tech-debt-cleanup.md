## The three debts

### Debt 1 — M9 recorder adapter missing

M9 backend `_start_voice` assumes:
```python
self._recorder = AudioRecorder(
    on_chunk_progress=self._on_recorder_chunk,
    on_auto_stop=self._on_recorder_auto_stop,
)
```

**Real `llm_code.tools.voice.AudioRecorder` API:**
- Constructor: `AudioRecorder(backend=None, sample_rate=16000, channels=1, *, silence_seconds=0.0, silence_threshold=3000)` — **no callback params**
- `.start()` → None
- `.stop() -> bytes` — returns captured PCM audio (not None)
- `.should_auto_stop() -> bool` — polled by UI timer
- `.elapsed_seconds() -> float`
- `._last_peak: int` (0-32767 range, 16-bit)
- `.stopped_no_speech: bool` / `.auto_stopped: bool` properties
- No `.transcribe()` method — transcription is a separate
  `STTEngine.transcribe(audio_bytes, language)` call

Result: in production, `AudioRecorder(on_chunk_progress=...)` raises
`TypeError: unexpected keyword argument`, caught by M9's try/except,
which prints `"voice unavailable: ..."`. **Voice does not work in
production v2.0.0.**

Tests pass because M9 e2e tests monkeypatch `AudioRecorder` with a
callback-compatible `FakeRecorder`.

### Debt 2 — `_screen_lock` is dead code

`ScreenCoordinator._screen_lock: asyncio.Lock` was introduced in M3
as the R1 risk mitigation — "single-writer invariant: PT redraw and
Rich print never interleave". But:

- M6 `LiveResponseRegion` explicitly **does not** acquire it (plan note:
  "Rich's Live has its own internal locking and M0 PoC showed no
  contention")
- M9 voice writes status via `update_status` → `_status_line.merge` →
  `app.invalidate()` — no lock acquisition
- The only caller of `async with self._screen_lock:` is
  `render_message_async`, which is not invoked from any production
  code path
- `acquire_screen()` is an unused async accessor

**Options:**
- (a) Actually use it — acquire it in every console.print path + every
  Live region update. Meaningful contention protection but adds
  complexity to hot paths that haven't shown problems.
- (b) Delete it — YAGNI. Rich + prompt_toolkit have shown no
  contention in M0 PoC or in 256 tests. If contention shows up, add
  it back with concrete evidence.

**Decision:** option (b) — delete. M0 PoC already proved that on
Warp/iTerm2/tmux the coexistence works without explicit locking.
Adding a lock without evidence of contention is speculative. If a
real race surfaces in M10+ testing, we re-add with a specific
reproducer.

### Debt 3 — R3 stress test is undersized

M9 `test_toggle_stress_no_deadlock` runs 10 iterations of
start/chunk/stop. Spec §10.1 R3 suggested 100. 10 was a CI-speed
compromise during M9 execution. It caught the basic deadlock risk
but doesn't exercise:
- Sustained background thread → main loop marshalling under real
  asyncio load
- Concurrent voice + streaming (voice active during a
  `LiveResponseRegion` render)
- Voice start immediately after voice stop (no cooldown)

Scale up from 10 to 100 iterations **and** add a concurrent
streaming/voice test.

**Design:**

```python
"""Polling-to-callback adapter for llm_code.tools.voice.AudioRecorder.

The real AudioRecorder exposes a polling API (should_auto_stop,
_last_peak, elapsed_seconds). The REPL backend wants a callback API
(on_chunk_progress, on_auto_stop) so background-thread events can
flow through asyncio.call_soon_threadsafe. This adapter bridges
the two: it wraps a real AudioRecorder, starts a polling asyncio
task on .start(), and emits callbacks at a fixed cadence until
.stop() is called or VAD auto-fires.
"""
from __future__ import annotations

import asyncio
from typing import Any, Callable, Optional

from llm_code.tools.voice import AudioRecorder


POLL_INTERVAL_SECONDS = 0.2
PEAK_NORMALIZE_DIVISOR = 32767.0   # 16-bit signed max


class PollingRecorderAdapter:
    """Bridge between AudioRecorder's polling API and M9 callbacks."""

    def __init__(
        self,
        *,
        on_chunk_progress: Callable[[float, float], None],
        on_auto_stop: Callable[[str], None],
        silence_seconds: float = 2.0,
        stt_engine: Any = None,
        language: str = "en",
    ) -> None:
        self._on_chunk = on_chunk_progress
        self._on_auto = on_auto_stop
        self._recorder = AudioRecorder(silence_seconds=silence_seconds)
        self._stt = stt_engine
        self._language = language
        self._audio_bytes: bytes = b""
        self._poll_task: Optional[asyncio.Task] = None
        self._started = False
        self._stopped = False

    def start(self) -> None:
        """Begin recording and schedule the polling task."""
        if self._started:
            return
        self._started = True
        self._recorder.start()
        # Schedule polling on the current event loop (the main one).
        loop = asyncio.get_running_loop()
        self._poll_task = loop.create_task(self._poll_loop())

    def stop(self) -> None:
        """Manual stop. Cancels the polling task, flushes recorder."""
        if self._stopped:
            return
        self._stopped = True
        if self._poll_task is not None and not self._poll_task.done():
            self._poll_task.cancel()
        self._audio_bytes = self._recorder.stop()

    async def transcribe(self) -> str:
        """Run STT on the captured audio. Returns transcript text."""
        if not self._audio_bytes or self._stt is None:
            return ""
        # AudioRecorder stopped_no_speech → bail early
        if self._recorder.stopped_no_speech:
            return ""
        # STT is sync — run on a thread so the main loop stays responsive
        return await asyncio.to_thread(
            self._stt.transcribe, self._audio_bytes, self._language,
        )

    async def _poll_loop(self) -> None:
        """Poll the recorder ~5× per second and emit callbacks."""
        try:
            while not self._stopped:
                await asyncio.sleep(POLL_INTERVAL_SECONDS)
                if self._stopped:
                    return
                elapsed = self._recorder.elapsed_seconds()
                peak_norm = min(1.0, self._recorder._last_peak / PEAK_NORMALIZE_DIVISOR)
                self._on_chunk(elapsed, peak_norm)
                if self._recorder.should_auto_stop():
                    # Auto-stop: snapshot audio, emit the callback, end
                    reason = (
                        "no_speech_timeout"
                        if self._recorder.stopped_no_speech
                        else "vad_auto_stop"
                    )
                    self._audio_bytes = self._recorder.stop()
                    self._stopped = True
                    self._on_auto(reason)
                    return
        except asyncio.CancelledError:
            pass
```

## Risks

### R1 — Adapter peak normalization math is wrong

`PEAK_NORMALIZE_DIVISOR = 32767.0` assumes peak is a 16-bit signed
value's absolute max. `AudioRecorder._last_peak` is set from
`numpy.abs(pcm_int16).max()` so peak is in [0, 32767]. Normalized to
[0.0, 1.0] via division. Verify by inspecting a real capture on a
quiet room (peak ~200 → ~0.006 normalized) vs speech (peak ~15000 → ~0.45).

Mitigation: snapshot-log the raw peak + normalized peak in a manual
smoke run. If the normalization is off, tune `PEAK_NORMALIZE_DIVISOR`.

### R2 — STT engine import / config loading

`create_stt_engine(config.voice)` is the factory in
`llm_code.tools.voice`. If `config.voice` is `None` (no voice config),
the factory may raise or return None. Adapter should treat `None` STT
as "no transcription" gracefully, which it already does
(`if self._stt is None: return ""`).

### R3 — Polling interval vs VAD responsiveness

`POLL_INTERVAL_SECONDS = 0.2` (5 Hz polling). Real VAD silence window
is typically 2.0s. 5 Hz means auto-stop is detected within 200ms of
the window elapsing. Acceptable latency.

If we drop to 10 Hz (0.1s) for snappier feedback, the poll task
runs 2× more but still consumes < 1% CPU. Decision deferred.

### R4 — `asyncio.to_thread` for STT is new to Python 3.9+

Our pyproject.toml requires `python_requires = ">=3.10"`, so
`asyncio.to_thread` is safe.
