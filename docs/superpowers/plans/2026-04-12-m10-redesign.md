# M10 Redesign — Dispatcher + Stream Renderer + AppState

> **Status:** Replaces `2026-04-11-llm-code-repl-m10-dispatcher.md`. The
> original M10 plan was written against a mental model of the codebase
> that turned out to be wrong in several important ways (invented
> `dispatcher.run_turn`, miscounted commands and file sizes, missed the
> app-state layer entirely). This doc captures the findings from a full
> reconnaissance of the v1.23.1 code and restates M10 on solid ground.

**Goal:** Split the TUI-coupled architecture of v1.23.1 into three
view-agnostic layers (AppState / ViewStreamRenderer / CommandDispatcher)
so M11 can flag-day cut over from `tui/app.py` to `cli/main.py +
view/repl/backend.py` cleanly, with every existing feature preserved.

**Key constraint:** No functional regression. Every command works. Every
streaming event (text, thinking, tool call, permission, compaction,
message stop) is rendered correctly. No commands deferred to v2.0.x.

---

## Why the original plan fell short

Reconnaissance of the actual code (run 2026-04-12 on `main`):

| Original plan assumption | Reality |
|---|---|
| `tui/command_dispatcher.py` is ~1500 lines, 62 commands | **2455 lines, 52 `_cmd_*` methods** |
| `COMMAND_REGISTRY` has 62 entries | **53 entries** |
| Dispatcher has a `run_turn` method that orchestrates LLM turns | **No such method.** `dispatch(name, args) -> bool` is a sync slash-command router. The actual turn orchestration lives in `tui/streaming_handler.py:33` — a 463-line `async def run_turn` that consumes `runtime.run_turn()`'s StreamEvent AsyncIterator. |
| "Widget access" is ~100-200 sites that need `render_message`/`update_status` swaps | **272 matches; only ~30 are real widget operations.** The other ~240 are `self._app._runtime` / `self._app._config` / `self._app._cost_tracker` / `self._app._skills` / `self._app._plan_mode` / ... — **application-level state**, not UI widgets. |
| Drop 4 legacy commands: `scroll`, `marketplace-browse`, `plugin-browse`, `settings-modal` | **None of these exist in the current registry.** The 62→58 math was against a hypothetical command list. |
| `test_tui/test_command_dispatcher.py` has ~150 tests | **22 tests.** The original-plan "~150 transliterated tests" target was unreachable because the source material doesn't exist. |
| Effort 8-12h | **Realistic 14-20h** given the missing layers. |

The original plan's 10.2 (ViewBackend extension) and 10.3 (dispatcher
rewrite) also silently assumed that `runtime` was the only dependency
dispatcher needed. In reality dispatcher threads 40+ state attributes
through `self._app._xxx`, and all of those have to live somewhere in
the new architecture.

## The key insight

`Runtime.run_turn()` (`runtime/conversation.py:762`) **is already
view-agnostic**. It produces an `AsyncIterator[StreamEvent]` with
typed events. The view-coupling problem only exists in
`StreamingHandler` (the consumer) and `CommandDispatcher` (which also
touches widgets when commands produce UI output).

**So the split we need is at the StreamEvent boundary, not at the
"rewrite the dispatcher" boundary.** Runtime produces events. A new
view-agnostic renderer consumes them. Dispatcher uses the renderer
for LLM turns and view methods for everything else.

## Invariants preserved from v1.23.1

1. **Runtime is untouched.** `runtime/conversation.py:Runtime.run_turn`
   signature and behavior stay the same. Zero runtime test changes.
2. **StreamEvent contract is the wire format.** New renderer consumes
   exactly the same 8 event types old streaming_handler did. No new
   event types. No changes to `api/types.py`.
3. **Every command that worked in v1.23.1 works in v2.0.0.** No
   command deferred. No command dropped.
4. **Permission dialog semantics unchanged.** v1.23.1 shows a select
   dialog with Allow / Always kind / Always exact / Edit / Deny; new
   renderer shows the same via `view.show_select`, powered by M8
   DialogPopover.
5. **Cost tracking and token counters** live on AppState (not on
   LLMCodeTUI, not on coordinator). Every code path that used
   `self._app._input_tokens += N` becomes `self._state.input_tokens += N`.

## Invariants **not** preserved (documented M10 changes)

1. **tui/command_dispatcher.py is NOT deleted yet.** It stays in place
   until M11 so the old TUI keeps working during development.
2. **tui/runtime_init.py's `RuntimeInitializer`** is kept as a thin
   adapter — its body moves to `runtime/app_state.py`, and
   `RuntimeInitializer.init()` calls `AppState.from_config()` internally.
   This way both old TUI and new CLI entry point go through the same
   factory. M11 deletes the thin adapter.
3. **`ViewBackend` Protocol grows one method: `request_exit()`.** Needed
   by `/exit`, `/quit`, Ctrl+D fallthrough, and the `_graceful_exit`
   flow. Implemented by REPLBackend (delegates to coordinator) and by
   StubRecordingBackend (sets `_running = False`).
4. **Voice state unification.** M9 put voice fields on `REPLBackend`
   (`_voice_active`, `_recorder`, `_loop`). M10 moves these to
   `AppState.voice_active`, `AppState.voice_recorder`, and the `_loop`
   capture stays on `REPLBackend` because it's backend-specific.
   `_cmd_voice` can then read/write through `state.voice_*`.

## Key design decisions

### 1. `AppState` is a dataclass, not a Protocol

Option A (Protocol): zero refactor to `tui/app.py`, dispatcher types
against the Protocol. **Rejected** — the M11 cutover needs to
instantiate AppState standalone (outside LLMCodeTUI), which requires a
concrete class with a constructor.

Option B (dataclass with factory): `@dataclass class AppState: ...` +
`AppState.from_config(config, cwd, ...)` classmethod. LLMCodeTUI
either uses `AppState` internally (invasive) or duck-types alongside
(fine until M11). **Chosen.**

Under this choice, `tui/runtime_init.py` is updated so its `init()`
method delegates to `AppState.from_config` and then copies fields
back onto `LLMCodeTUI`. Thin adapter, preserves old TUI behavior.

### 2. `ViewStreamRenderer` in `view/`, not `view/repl/`

The renderer takes a `ViewBackend` and works for any backend. Putting
it in `view/repl/` would couple it to one backend. **Put in `view/`.**

### 3. `run_turn` on `CommandDispatcher`, not `ViewStreamRenderer`

`ViewStreamRenderer.run_turn(text, images)` does the stream rendering.
`CommandDispatcher.run_turn(text)` is the **top-level entry point** —
it parses slash commands, handles custom commands + skill commands,
and for plain text it awaits `renderer.run_turn(text)`. The backend's
`set_input_handler(dispatcher.run_turn)` wires the whole thing together.

This matches M11's `backend.set_input_handler(dispatcher.run_turn)`
expectation at Task 11.2.

### 4. Permission dialog routing

v1.23.1 `StreamingHandler` calls `self._app._dialogs.select(...)` — a
Textual-specific dialog system. v2.0.0 `ViewStreamRenderer` calls
`await self._view.show_select(prompt, choices, default="allow")` — the
M8 `DialogPopover` handles it uniformly.

The 5 choices (Allow / Always kind / Always exact / Edit / Deny) stay.
The "Edit args" branch opens a text input via
`await self._view.show_text_input(...)` then sends the edited args
back via `runtime.send_permission_response("edit", edited_args=parsed)`.

### 5. `on_voice_toggle` rewiring

M9 backend owns `_toggle_voice` and the recorder lifecycle. M10
preserves this — dispatcher's `_cmd_voice` still has a sub-command
structure (list/start/stop/...), but for the toggle path
(Ctrl+G), the backend's existing wiring is untouched. M10 just moves
the `voice_active` / `voice_recorder` fields from REPLBackend into
AppState so dispatcher can read them.

Specifically:
- Backend's `_toggle_voice`, `_start_voice`, `_stop_voice`,
  `_on_recorder_chunk`, `_on_recorder_auto_stop`,
  `_transcribe_and_insert` all remain on `REPLBackend`.
- But `_voice_active`, `_recorder` move to `AppState` and backend
  accesses them via `self._app_state.voice_active`. This requires
  backend to hold a reference to AppState — added in 10.3.

### 6. `cli/main.py` stays out of M10 scope

M11 Task 11.2 is where `cli/main.py` gets written. M10 leaves
`cli/tui_main.py` in place so the current entry point keeps working.
M10 ends with both systems alive: old (LLMCodeTUI) and new
(REPLBackend + CommandDispatcher + ViewStreamRenderer + AppState).
M11 is the flag day that swaps entry points and deletes tui/.

## Risks

### R1 — AppState field drift during the LLMCodeTUI refactor
If we touch `tui/app.py` to use `AppState` internally, we risk
breaking the old TUI tests (there are ~500 of them). Mitigation: M10
does NOT modify `tui/app.py`. The thin adapter in
`tui/runtime_init.py` keeps old TUI fully functional. Any divergence
between `LLMCodeTUI.__init__` and `AppState.from_config` is a code
smell handled at M11 (delete tui/).

### R2 — StreamEvent semantics drift
The renderer port is ~500 lines of careful iteration over 8 event
types. If any branch's behavior changes (e.g., tool_id tracking, stop
reason capture, empty-response diagnostic), the tests catch it.
Mitigation: **Transliterate the tests first or in lockstep with the
renderer port**, don't skip to verification at the end.

### R3 — Permission dialog reentry
v1.23.1's `StreamingHandler` awaits `self._app._dialogs.select(...)`
while the stream is still emitting events. In asyncio that's fine
because the await yields to the event loop. But our M8
`DialogPopover.show_select` uses `asyncio.get_running_loop().create_future()`
— that loop has to be the same loop the stream is running on. It is
(everything runs on the single asyncio loop M9's voice path captured),
but verify with a test that drives a PermissionRequest through a real
`REPLBackend` pilot.

### R4 — `AppState.from_config` duplication
If we copy the body of `RuntimeInitializer.init()` into
`AppState.from_config`, any future runtime-init change has to happen
twice. Mitigation: **move, don't copy**. `RuntimeInitializer.init()`
becomes `state = AppState.from_config(...); self._app.X = state.X ...`
— a field-copy loop. When M11 deletes tui/, the adapter goes with it
and AppState is the single source of truth.

### R5 — Test count explosion
The new `test_dispatcher.py` will transliterate ~22 old tests and add
~30 new ones (one per `_cmd_*` for happy-path coverage). Combined with
`test_stream_renderer.py` (~55 tests), that's 100+ new tests in test_view/.
The current `test_view/` runs in ~1 second; adding 100+ more tests,
some of them exercising async state machines, could push it to ~5
seconds. **Acceptable.**

## Fallback

None — this redesign **has no fallback**. If 10.3-10.7 overrun, the
response is to continue, not to ship partial. The original F3 fallback
(keep tui/command_dispatcher.py in place) is explicitly rejected
because it breaks the M11 flag day and leaves the codebase in a
two-dispatcher state that would have to be cleaned up anyway.

If M10 is genuinely blocked by a problem this design didn't foresee,
the response is to update this doc and re-plan, not to half-do it.

## Next milestone: M11 — Cutover

After M10 completes, M11 Task 11.2 rewrites `cli/main.py` to construct:

```python
state = AppState.from_config(config, cwd)
backend = REPLBackend(config=config, app_state=state)
renderer = ViewStreamRenderer(view=backend, state=state)
dispatcher = CommandDispatcher(view=backend, state=state, renderer=renderer)
backend.set_input_handler(dispatcher.run_turn)
asyncio.run(backend.run())
```

and then deletes `tui/` entirely.
