# M11-M14 Plan Audit — Drift From Real Code

> **Status:** Supplement to M11-M14 plans. Does NOT replace them. Each
> plan still drives its own execution, but this doc flags places where
> the plans assume facts that aren't true in the v1.23.1 codebase (or
> conflict with the M10 redesign committed 2026-04-12). **Read this
> before executing any of M11/M12/M13/M14.**

**Context:** M10 reconnaissance revealed the original M10 plan was
written against a mental model that diverged significantly from the
real code. Rather than trusting M11-M14 blindly (which were written in
the same sitting), this doc audits each one against reality.

**Methodology:** Read each plan end-to-end, check every concrete claim
(file paths, command counts, test counts, env vars, method names) via
grep / file inspection / pytest collect, record divergences.

### 🔴 H2. Task 11.2 `cli/main.py` template uses old REPLBackend constructor + invented Runtime method

Plan template:
```python
runtime = Runtime(config=config, cwd=cwd)
backend = REPLBackend(config=config, runtime=runtime)
dispatcher = CommandDispatcher(view=backend, runtime=runtime)
backend.set_input_handler(dispatcher.run_turn)
runtime.on_status_change(backend.update_status)
```

Two problems:
1. **`runtime.on_status_change`** does not exist in
   `llm_code/runtime/conversation.py`. `grep -n "on_status_change"
   llm_code/runtime/conversation.py` returns 0 matches. Plan invented
   this method.
2. **M10 redesign (2026-04-12) introduces `AppState`** — the correct
   construction pattern is:

```python
state = AppState.from_config(config, cwd=cwd, budget=budget, initial_mode=mode)
backend = REPLBackend(config=config, app_state=state)
renderer = ViewStreamRenderer(view=backend, state=state)
dispatcher = CommandDispatcher(view=backend, state=state, renderer=renderer)
backend.set_input_handler(dispatcher.run_turn)
asyncio.run(backend.run())
```

No `runtime` is instantiated standalone. `state.runtime` is built by
`AppState.from_config`. No `on_status_change` call — the renderer
pushes status updates directly via `view.update_status` during the
stream loop.

**Fix:** Rewrite M11 Task 11.2 code template to match the M10 redesign
output. Drop the `on_status_change` line entirely.

## M12 — Pexpect Smoke Tests (2 high, 1 med)

### 🔴 H3. `LLMCODE_TEST_MODE` / `LLMCODE_NO_TELEMETRY` env vars don't exist

Plan Task 12.2 conftest:
```python
env["LLMCODE_TEST_MODE"] = "1"
env["LLMCODE_NO_TELEMETRY"] = "1"
```

`grep -rn "LLMCODE_TEST_MODE\|LLMCODE_NO_TELEMETRY" llm_code/ tests/`
returns 0 matches. Neither env var is wired to anything. Setting them
has no effect — the spawned `llmcode` will still try to call a real
LLM and real telemetry endpoints.

**Fix options:**

- (a) **Add `LLMCODE_TEST_MODE=1` support** in `cli/main.py` — when
  set, it wires a `MockLLMProvider` that echoes `"echo: <input>"` for
  every turn instead of calling a real LLM. ~30 lines, fits into M10
  Task 10.3 (AppState).
- (b) **Use a real but local mock server** — spawn a pytest fixture
  that runs a tiny HTTP server speaking the OpenAI chat API, point
  `LLMCODE_API_BASE` at it. More realistic but more setup.
- (c) **Skip LLM-invoking tests** — M12's golden path tests (version,
  help, quit, slash popover, Ctrl+D, etc.) don't need an LLM. Only
  `test_streaming_fake_response_visible` does. Drop that one test
  or put it behind `pytest.mark.skipif(not os.environ.get("LLMCODE_REAL_API"))`.

**Recommendation:** Option (a). Add `LLMCODE_TEST_MODE` as a real
feature in M10 Task 10.3 (AppState) so M12 can rely on it. It's also
useful for local development — `LLMCODE_TEST_MODE=1 llmcode` lets
you eyeball the UI without burning API tokens.

### 🔴 H4. Several M12 tests depend on behaviors that haven't been implemented yet

- `test_slash_popover_appears_on_slash` — needs `pexpect.expect(r"voice|version|vim", timeout=2)`. Works only if the slash popover renders visible text in the PT output. M4's `SlashCompleter` produces PT `Completion` objects; whether they surface in `pexpect.before` depends on PT's rendering to the TTY. **Verification needed**: write a smoke test early in M12 that literally captures what PT sends to the terminal when `/` is typed, and assert on reality rather than the plan's assumption.
- `test_streaming_fake_response_visible` — depends on LLMCODE_TEST_MODE (H3)
- `test_history_recall_with_ctrl_up` — sends `\x1b[1;5A` (xterm Ctrl+Up). pexpect doesn't always render what's in the buffer until submit, so the test works around it by submitting and checking /version runs again. Fragile but OK.

**Fix:** M12 Task 12.3 Step 1 should be a "pexpect baseline capture" — spawn llmcode, type a few characters, capture EVERYTHING that comes back, write a golden. This calibrates all subsequent tests.

### 🟡 M3. M12 `pexpect.spawn` uses hardcoded binary path

Plan conftest:
```python
LLMCODE_BIN = Path("/Users/adamhong/miniconda3/bin/llmcode")
if not LLMCODE_BIN.exists():
    LLMCODE_BIN = Path(shutil.which("llmcode") or "llmcode")
```

Works for Adam's dev env but not for CI or any shared dev setup.

**Fix:** Prefer `shutil.which("llmcode")` first, fall back to the
hardcoded path. Or use `sys.executable + " -m llm_code.cli.main"` to
avoid the whole "is `llmcode` on PATH?" question. The
`sys.executable -m` form is more portable for pexpect and always
available.

## M13 — Snapshot Tests (0 high, 0 med, 2 low)

### 🟢 L3. Dialog snapshot tests touch `popover._active` directly (white-box)

Plan Task 13.2 builds `ConfirmRequest(future=loop.create_future(), ...)`
and assigns it to `popover._active` directly, bypassing the async
`show_confirm` flow. This is intentional to avoid making snapshot
tests async (the render can be captured from static state).

**Acceptable** but should be explicit in the plan doc. **Fix:** Add a
one-line comment in the plan acknowledging this is a deliberate
white-box technique, not an API misuse.

### 🟢 L4. Spinner non-determinism in status line snapshot

`test_snapshot_status_line_streaming` assumes `_spinner_frame = 0` at
render time. StatusLine initializes `_spinner_frame = 0` so this is
stable for a fresh instance, but the assumption is fragile —
`coordinator._status_text` calls `advance_spinner()` on every render,
so anything that renders the status line once before capture will
shift the frame.

**Fix:** The snapshot test should always construct a fresh
`StatusLine()` and NOT call `advance_spinner()` between setup and
capture. Plan already does this. Just add a guard comment:
`# NB: spinner_frame is 0 by construction; do not advance before capture`.
